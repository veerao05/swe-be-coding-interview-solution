Spring Boot Interview Cheat Sheet

List of possible questions

Service Layer

IssueQuestion They'll AskOne-line Answer
Duplicate conversion logic in 3 methods "How do you eliminate the duplication? What's wrong with forEach?" Extract toActivityDto() helper; use stream().map() — no side effects
findAll() then filter in memory "What's wrong? What happens with 10 million rows?" Loads all rows into memory — use findById() or JPQL WHERE clause
Dead code — searchActivities() always returns [] "What does this method actually return?" Items added to activitiesDto but result (empty) is returned — classic copy-paste bug
Inconsistent null handling — NPE in 2 of 3 methods "What happens if supplier is null? When does it actually crash?" @NotFound(IGNORE) + orphaned supplier_id=5 in seed data guarantees NPE at runtime
Missing @Transactional(readOnly = true) "Why use readOnly = true?" Hibernate skips dirty checking — better performance for read-only queries
N+1 query problem "How many queries for 100 activities? How do you fix it?" 101 queries (1 + N lazy supplier loads) — fix with LEFT JOIN FETCH a.supplier
Checked exception in lambda "Why doesn't ActivityNotFoundException extends Exception compile in a lambda?" Checked exceptions can't be thrown from functional interfaces — extend RuntimeException
Statistics query returns SELECT * "Does getSupplierStats() return actual statistics? What's wrong with List<Object[]>?" No — it's just SELECT *. Object[] is type-unsafe. Use JPQL COUNT/AVG/SUM GROUP BY with a DTO

Controller Layer

IssueQuestion They'll AskOne-line Answer
EntityManager directly in controller "Why shouldn't you put EntityManager in a controller?" Violates layered architecture — data access belongs in the repository, not the web layer
In-memory supplier search with wrong fallback "Is this search case-sensitive? What does it return when nothing matches?" Case-sensitive; returns ALL suppliers as fallback — use JPQL LIKE LOWER() instead
@Controller instead of @RestController "What's the difference? Will this work?" @Controller resolves view names — needs @ResponseBody per method. @RestController adds it to all methods
@PathVariable null check "When would id ever be null in a @PathVariable?" Never — Spring returns 400 before the method is reached. The null check is dead code
Catch-all error handler always returns 500 "What's wrong with one @ExceptionHandler(Exception.class) handler?" A 404 becomes a 500. Use specific handlers per exception type with correct HTTP codes

Entity Layer

IssueQuestion They'll AskOne-line Answer
@Data on JPA entity "Why is @Data dangerous on a JPA entity?" toString() and equals() touch lazy proxies → LazyInitializationException or StackOverflowError
Bidirectional toString() recursion "What happens when Supplier calls toString() on its activities?" Infinite recursion: Supplier→Activity→Supplier→... → StackOverflowError. Fix: @ToString.Exclude
@NotFound(action = IGNORE) "What does @NotFound(IGNORE) do when the supplier row doesn't exist?" Hibernate silently sets supplier = null instead of throwing — callers must null-check
GenerationType.AUTO "What's the difference between AUTO and IDENTITY?" AUTO may use a hibernate_sequence table; IDENTITY always uses the DB identity column
Primitive vs wrapper types "Why use Double instead of double in an entity?" Primitives can't be null — wrapper types map correctly to nullable DB columns

Repository & DTO Layer

IssueQuestion They'll AskOne-line Answer
JPQL constructor expression "Why does SELECT new SupplierStats(...) fail without @AllArgsConstructor?" JPQL calls the constructor by position — no @AllArgsConstructor means no matching constructor
Field order in DTO vs JPQL "What happens if field order in the DTO doesn't match the JPQL select?" Silent wrong values — no compile error, no runtime exception. totalRevenue gets averageRating's value
SupplierStats in wrong package "Where should a DTO class live?" In dto package — not repository. Mixing them breaks separation of concerns
Native SQL vs JPQL "When would you use native SQL over JPQL?" JPQL is DB-portable and type-safe. Native SQL only when DB-specific features (window functions, hints) are needed

Testing

IssueQuestion They'll AskOne-line Answer
@WebMvcTest vs @SpringBootTest "What's the difference? When do you use each?" @WebMvcTest — web layer only, mocked service, fast. @SpringBootTest — full context, real DB, slow
@DataJpaTest with replace = NONE "Why add @AutoConfigureTestDatabase(replace = NONE)?" Without it Spring replaces your configured datasource with an embedded one — your Flyway/SQL scripts may not run
Original tests called controller directly "What's wrong with calling controller.getActivities() in a test?" Bypasses HTTP layer — no status codes, no serialization, no filters. Use MockMvc.perform(get(...))
Original tests had dead mock (SupplierController) "Why mock something that's never used in the test?" Dead mocks are noise — they don't fail but mislead reviewers and hide missing test coverage
